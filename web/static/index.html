💡 [互斥] usage（main/planner/reviewer/extractor）一次只能被一个 model 占用—— 勾上时若已有其他 model 占用，会弹窗确认是否替换。后端 PUT 时也会强制 enforce （自动从旧 model 上移除该 usage）。

💡 main vs planner vs extractor — 三类任务模型需求完全不同，建议分开：
  • main：writer 写章正文。要长文本 + 文笔（Claude Sonnet / DeepSeek V4 Pro）
  • planner：所有"先想后写"的规划 agent。要推理 + JSON（可比 main 更强或更便宜，看你侧重）
  • extractor：从文本抽 JSON。短输入短输出，轻量便宜模型够（gemini-flash / haiku）
  • reviewer：审核打分。快速便宜（gemini-flash）

💡 没绑的 usage 自动 fallback 到 main——向后兼容，但实际能力 / 成本都不如显式绑。

💡 in_story_ai：SpecialAbility.external_llm_profile 下拉框会过滤显示 勾过此 usage 的 profile，避免选错主写作模型导致 DeepSeek/Claude 等元语言污染正文。
💡 同一用途有多条时取第一条匹配（要换就删掉旧的或在编辑里取消勾选）。
  • 勾 main + reviewer：一条记录同时承担主与审核两用途，省一组 key
  • 单独勾 fallback：主调用 120s 无响应或全部重试失败时切到这条；fallback 也失败才向上抛错
  • 全部不勾：这条记录只是"登记/备份"在那里，运行时不会被路由到——以后想用直接编辑勾上即可
